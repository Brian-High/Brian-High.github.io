Conway's Game of Life

Personal Project

Java  ·   JavaFX
Memory Optimization  ·  Run-Length Encoding  ·   File I/O  ·  Complexity Theory

Objective:

Emulate the famous cellular automata Conway's Game of Life. This is the second attempt I made at this project and it was greatly improved in the second version. I became interested in this topic because of it's relation to complexity theory - how complex structures can emerge from simple rules. If you are unfamiliar, the rules can be found here.

Program Design:

The program is a 2D grid of cells that can be turn on or off and a control bar. The visual part of the part is a canvas that the program draws onto using a coordinate system. The user can use the mouse to create a pattern or import a pattern from this website. Users can pause or play the simulation, changes the speed the simulation runs, clear the board, or randomize the simulation (turning random cells on).

Run-Length Encoding (RLE)

Run-Length Encoding is a popular compression method that is useful when data has long stretches of data that is the same. For example, “wwwwaaadexxxxxx” would be encoded as “w4a3d1e1x6”. This format works well when exporting board states from this program because there are only two states for each cell, and there are typically long stretches of the same state. The best part of the Game of Life is the large and complex patterns that can do various things. In order run these patterns I created a decoder for the rle format so that I could import files from this website.

What I Learned:

The original version of this program was slow and inefficient at large scales (100 x 100 and larger boards). I used a tool called Visual VM to analyze the performance of the program and I identified memory to be the limiting factor. The reason why the program was using so much memory was because I was storing the state and position of each cell along with the visual component. The new version of the program only stored the position and state and the visual component drawn onto a canvas. By switching out a large array of buttons with a canvas and not storing the unnecessary visual data, the new version of the program became much faster.
